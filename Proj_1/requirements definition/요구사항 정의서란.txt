요구사항 정의서란
1. 우리회사는 이런 주제로 서비스 론칭하려고하는데, 
   이런 기능과 저런 기능들은 포함되어야해!!
2. 전달 받은 Request For Proposal 를 토대로 
   영엽팀, 기획자 , PM 이 제안서를 작성한다.
3. 새로운 기능을 요구하는 고객사와 트러블이 마구마구 생기게 된다.
4. 100% 완벽한 요구사항 정의는 없습니다.



요구사항 정의서 작성하는 방법
- 요구사항 구분
   -> 웹, 앱, 사용자, 관리자, 공통

- 요구사항 ID 
   -> 커뮤니케이션이 덜 혼란스럽고 명확하기 때문에 사용.

- 요수사항 명 
   -> 어떤 요구사항인지 요약하며 기제(명사형)으로 작성함.  

- 요구사항 상세 설명__기능 
   -> 고객 요구사항에 대해 자세히 파악했는지 확인 작업.

- 요청자__이름 
   -> 요구사항에 대한 자세한 논의가 필요한 상황 대처용.

- 비고__이슈 
   -> 레퍼런스를 전달 받거나, 확인 요청을 할 때 사용.



이외에 갈럼에 따라 요구사랑을 작성할 수 있다.
- 메뉴 위치 __ [ 메인 -> 고객센터 -> 자유게시판 ]
   -> 메뉴 구조도IA와 요구사항 정의서를 문서로 작성가능.

- 중요도__ [상/중/하],[Major/Minor]
   -> 우선순위 파악을 위해 담당자의 판단이 핵심!!

- 요청 일자__[08/05],[2021년 08월 05일]
   -> 요청 일자 기재용.
   -> 온갖 메일을 다 뒤져 보지 않아도 됨.

- 수행사 수행/수용 여부__[수행],[협의 필요],[협의 진행 중]
   -> 디자인/개발팀에 수행 여부를 판단_PM
   -> 수행 여부를 명확히 전달해주는 작업이 필요

- 난이도__[상/중/하],[매우 높음]
   -> 고객사와 일정 조정을 위한 질문용
   -> 작업 시간에 따라 난이도를 작성한다.

- 요구사항 근거__ [유선],[이메일],[08/05 고객사 회의]
   -> 어떤 통화를 통해 전달받은 요구사항 기제
   -> 전달 받은 히스토리를 남겨두는 것이 도움이 됨.
